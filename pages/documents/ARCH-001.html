ARCH-001 — Current Studio Architecture

Two Marshalls Studios — Studio Operating System (TMS-OS)

1. Purpose

This document defines the verified current architecture of TMS-OS and the operating layers that allow studio data, shared engines, department pages, and permanent documentation to work together.

2. Current Architecture

Architecture State: Fully Modular StudioDB

TMS-OS separates structured records from the code that renders them. Record files define studio information, loader.js registers those records, StudioDB provides shared access methods, and page-specific renderers display the approved information.

3. Operating Layers

Presentation Layer

HTML pages and shared CSS provide the interface for Headquarters and studio departments.

Shared Layout Layer

layout.js provides the shared header, global navigation, and footer.

Renderer Layer

app.js and department-specific renderer files transform StudioDB records into visible content.

Shared Component Layer

components.js provides reusable interface components across departments.

StudioDB Access Layer

loader.js assembles records and exposes query, validation, search, and export functions through window.StudioDB.

Modular Record Layer

Files under js/studio-data/records store structured studio records separately from rendering code.

4. Documentation Architecture

Permanent documents are registered as a StudioDB record type. The Documentation Catalog Engine displays those records, and the TMS Document Viewer Engine loads approved internal content files without placing document bodies inside documents.js.

5. Source-of-Truth Boundaries

GitHub preserves source code and project history. Google Headquarters preserves permanent human-readable governance documents. StudioDB provides the structured operating data used by the portal.

6. Current Verification

The dashboard, global navigation, shared search, Documentation Catalog, grouped catalog, and first embedded viewer connection have passed controlled functional testing during the Version 1.0 completion audit.

7. Permanent Statement

TMS-OS uses a modular architecture so data, rendering, interface components, permanent documentation, and future automation remain separated, testable, and maintainable across workstations.
